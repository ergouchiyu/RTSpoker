# Card RTS 商业项目开发计划

## 项目概述

**游戏名称：** Card RTS（暂定）
**游戏类型：** Real-Time Strategy × Poker
**核心卖点：** 一边打扑克，一边实时派兵偷袭对方的手牌
**目标平台：** PC (Steam) → Mobile (iOS/Android) → 主机

---

## 开发哲学

### 第一原则：先验证，再扩展

> "只有步兵、工程师、维修、牌产经济、出牌胜利"这一套循环，玩家是否愿意连续玩 10 局？
> 
> 如果答案是"愿意"，说明抓住了核心；之后再加内容，只是给一棵已经长成的树添枝加叶。
> 
> 如果答案是"不愿意"，说明需要优先调整核心循环，而不是靠更多兵种和更华丽的表现去掩盖问题。

### 第二原则：每增加一个系统，都必须让"出牌"和"RTS"之间的联系更紧密

---

## 技术选型

### 原型阶段（Phase 0）

**推荐：HTML5 + JavaScript + Canvas**

原因：
- 你已经熟悉 JavaScript（family-reward 项目）
- 浏览器运行，无需安装，方便测试和分享
- 快速迭代，改代码立即看到效果
- 后续可以轻松部署到网页供多人测试

### 商业阶段（Phase 1+）

**推荐：Unity (C#)**

原因：
- 跨平台（PC/iOS/Android/主机）
- 成熟的 2D 工具链
- 丰富的 Asset Store
- 大量学习资源和社区支持
- 商业游戏首选引擎

## 开发阶段

### Phase 0：核心玩法验证（2-3 周）

**目标：** 验证"扑克+RTS"这个核心循环是否有趣

**技术栈：** HTML5 + JavaScript + Canvas

**功能范围：**
- [ ] 扑克桌 UI（最简版）
- [ ] 发牌系统
- [ ] 手牌显示（自己看牌面，对手看牌背）
- [ ] 出牌规则（基础版，不做斗地主）
- [ ] 经济系统（手牌产资源）
- [ ] 步兵（自动寻路攻击对手的牌）
- [ ] 维修系统（恢复牌的完整度）
- [ ] 胜利条件（最先出完牌）

**不做：**
- ❌ 工程师（偷牌）
- ❌ 精美 UI/动画
- ❌ 联机
- ❌ AI
- ❌ 音效

**验收标准：**
- 本地 2 人对战（分屏或热座）
- 测试者愿意连续玩 10 局以上
- 核心循环清晰：留牌赚钱 → 出牌推进 → 派兵骚扰

---

**技术栈：** Unity (C#)

**功能范围：**
- [ ] 迁移到 Unity

- [ ] 完整度系统（HP）
- [ ] 人口系统
- [ ] 基地系统（生产/维修/升级）
- [ ] 基础 AI（单人模式）
- [ ] 数值平衡调整
- [ ] 基础 UI 美化

**验收标准：**
- 单人 vs AI 可玩
- 所有核心机制正常运作
- 数值基本平衡

### Phase 2：多人联机（4-6 周）

**目标：** 实现在线对战

**功能范围：**
- [ ] 网络架构（客户端-服务器）
- [ ] 房间系统
- [ ] 匹配系统
- [ ] 同步机制（扑克状态 + RTS 状态）
- [ ] 断线重连

**技术选型：**

### Phase 3：内容扩展（4-6 周）

**目标：** 增加游戏深度和可玩性

**功能范围：**

- [ ] 不同地图（不同牌桌布局）

**验收标准：**

## 数据结构设计（Phase 0）

### 扑克牌

```javascript
class Card {
    constructor(suit, rank) {
        this.suit = suit;        // ♠ ♥ ♦ ♣
        this.rank = rank;        // 2-10, J, Q, K, A
        this.hp = 100;           // 完整度
        this.maxHp = 100;        // 最大完整度
        this.income = 2;         // 基础收入
        this.position = { x: 0, y: 0 }; // 位置
        this.owner = null;       // 所属玩家
    }
}
```

### 玩家

```javascript
class Player {
    constructor(id) {
        this.id = id;
        this.hand = [];          // 手牌
        this.supply = 0;         // 补给
        this.population = 0;     // 当前人口
        this.maxPopulation = 10; // 最大人口
        this.units = [];         // 部队
    }
}
```

### 单位

```javascript
class Unit {
    constructor(type, owner) {
        this.type = type;        // 'infantry' | 'engineer'
        this.owner = owner;
        this.hp = 100;
        this.attack = 5;         // 攻击力
        this.speed = 2;          // 移动速度
        this.position = { x: 0, y: 0 };
        this.target = null;      // 目标
    }
}
```

## 数值平衡（Phase 0）

| 项目 | 初始值 | 说明 |
|------|--------|------|
| 基础人口 | 10 | 限制部队数量 |
| 每张牌基础收入 | 2/s | 经济来源 |
| 步兵价格 | 100 | 基础战斗单位 |
| 工程师价格 | 300 | 高级单位（Phase 1） |
| 步兵训练时间 | 10s | 生产节奏 |
| 卡牌最大完整度 | 100 | HP 上限 |
| 步兵攻击 | 5 HP/s | 攻击速度 |
| 维修速度 | 2 HP/s | 恢复速度 |
| 维修费用 | 2 补给/s | 维修成本 |

**设计目标：** 单个步兵可以压制维修，但不能轻易摧毁牌；需要多个单位协同才能快速拆掉关键牌。

## 风险与挑战

### 技术风险

1. **网络同步** - 扑克+RTS 双重同步难度高
2. **性能优化** - 大量单位同时运行
3. **防作弊** - 实时游戏容易被破解

### 设计风险

1. **核心循环是否有趣** - 需要大量测试
2. **数值平衡** - 扑克和 RTS 的平衡
3. **学习曲线** - 两种玩法叠加可能太复杂

### 商业风险

1. **市场定位** - 小众品类
2. **竞品分析** - 需要差异化
3. **留存率** - 核心玩法是否足够持久
